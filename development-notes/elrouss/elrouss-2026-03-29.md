# 29.03.2026

### ✅ Что сделано:

1. Убрал запуск пользовательского кода через `new Function` в воркере;
2. Подключил **QuickJS (quickjs-emscripten)** в отдельном **Web Worker**: и **Run** (консоль), и **Test** (вызов решения + сравнение с ожиданием) идут внутри изолированной VM, а не в основном потоке;
3. Разрулил загрузку **`.wasm`**: в дев-сборке Emscripten тянул файл по неверному URL и получал HTML вместо бинарника — вынес `emscripten-module.wasm` в **assets** и задал **`wasmLocation` через `newVariant`**, чтобы путь был предсказуемый;
4. Ввёл служебный тип сообщения **`done`** от воркера, чтобы отличать «закончился прогон» от **`success`** по отдельному тесту и вовремя снимать таймаут/спиннер;
5. Вынес «заглушки-колбэки» для тестов из кода воркера в **данные задачи**: поле **`builtinFns`** в payload (пока в JSON-моке), в `args` остаётся ссылка **`{ "$fn": "…" }`** — чтобы не плодить одни и те же строки функций в каждом кейсе и не ломать JSON;
6. Поправил мок: неверное **ожидаемое значение** во втором тесте (`11` вместо `12` при `val + i + arr.length` для `[10]`).

### ❌ Проблемы:

1. Попробовал промежуточный вариант с **sandbox iframe** (`srcdoc` + CSP + handshake), но он плохо работал в dev (гонки `postMessage` / путь к ресурсам) — на нем я потерял время, зато четче понял, где граница «изоляция в iframe» vs «интерпретатор в WASM»;
2. Документация по **quickjs-emscripten** размазана по репозиторию: без чтения про **`newVariant` / wasmLocation** легко упереться в бессмысленную ошибку «magic word wasm»;
3. **Абсолютной** безопасности на фронте всё равно нет: QuickJS режет доступ к DOM/`fetch`, но DoS по CPU остаётся — я оставил прерывание по дедлайну + `terminate` воркера по таймауту с родителя.

### 🧠 Решения, мысли и планы:

В продолжение темы с **Monaco** и «редактор даёт только строку» я наконец закрыл для себя вопрос **как это исполнять**:

1. **Почему не оставить `Function` в воркере**  
   Воркер решает фриз UI, но не даёт изоляции уровня «чужой код не видит браузерные API». Плюс ESLint и мысли о безопасности давят на отказ от `eval`-семейства в основном коде приложения. Хотелось хотя бы **осознанный компромисс**: код крутится в **отдельной JS-машине (WASM)**, а не в том же контексте, что и приложение.

2. **Iframe с CSP vs QuickJS**  
   Iframe без `allow-same-origin` + жёсткий CSP — рабочая схема «как у многих песочниц», но в связке с Angular dev server и динамическими чанками я поймал неприятные моменты с доставкой и порядком сообщений. **QuickJS** оказался тяжелее по весу бандла, зато **один и тот же** путь и для логов, и для тестов, и предсказуемая модель: строка → VM → `postMessage` наружу. Для MVP учебного проекта это выглядит более контролируемо, чем дальше отлаживать iframe.

3. **Почему в `args` не «лежит функция»**  
   С бэка приходит **JSON**. Функции в JSON не живут — только строки/объекты. Варианты я для себя разложил так:  
   - либо **в каждом тесте** тащить исходник колбэка строкой (дубли, сложнее ревьюить);  
   - либо **`builtinFns`** один раз на задачу + **`$fn`** в аргументах (как сейчас) — явный реестр «что разрешено подставлять в тесты». Для бэка это тоже проще описать в OpenAPI: одна карта + массив кейсов.
