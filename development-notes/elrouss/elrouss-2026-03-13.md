### 🧠 Решения, мысли и планы:

В первой половине недели сделал запланированную половину пулла работ:
1. Разобрался c хлебными крошками в Ангуляре. Я думал, что в фреймворке будет какое-то готовое решение, однако, это оказалось не так. Перечитал множество статей на данную тему. В итоге, подошло решение со Stack Overflow и из статьи на Medium. Интегрировал и адаптировал их под нужды нашего проекта. Если говорить о решении, то суть его сводится в первую очередь к древовидному построению роутинга. Изначально построил так, как было указано на Stack Overflow:

```ts
  {
    path: 'quiz',
    data: { breadcrumb: 'Quiz' },
    component: CategoriesPageComponent,
    children: [
      {
        path: ':categoryId',
        data: { breadcrumb: categoryBreadcrumbResolver },
        component: CategoryPageComponent,
        children: [
          {
            path: 'topic/:topicId',
            data: { breadcrumb: topicBreadcrumbResolver },
            component: QuizPageComponent,
          }
        ]
      },
    ]
  }
```

Выглядит логично, но такая схема оказалась не работоспособной из-за `Outlet`. Angular не делал перерисовку страницы при входе на второй уровень глубины (сущность топика квиза).

Нашел примеры, где описана данная проблема и по итогу привел к текущему формату:

```ts
  {
    path: 'quiz',
    data: { breadcrumb: 'Quiz' },
    children: [
      {
        path: '',
        data: { breadcrumb: null },
        component: CategoriesPageComponent,
      },
      {
        path: ':categoryId',
        resolve: { breadcrumb: categoryBreadcrumbResolver },
        children: [
          {
            path: '',
            data: { breadcrumb: null },
            component: CategoryPageComponent,
          },
          {
            path: 'topic/:topicId',
            resolve: { breadcrumb: topicBreadcrumbResolver },
            children: [
              {
                path: '',
                data: { breadcrumb: null },
                component: QuizPageComponent,
              },
            ]
          }
        ]
      }
    ]
  }
```

Выяснилось, что для Ангуляра крайне важно вкладывать ссылку на компонент родительского роута в дочерний с пустой строкой. Далее написан просто сервис, который рекурсивно обходит данное дерево и собирает указанные в `breadcrumb` значения в массив, который и отрисовывается в шаблоне. Также важно явно прописывать `data: { breadcrumb: null }`, для того чтобы в рекурсию не попадали дубли в виде "хлебной крошки" из родителя. Из очень интересного - это динамические параметры в урле (`:categoryId`, `:topicId`): к ним в Ангуляре можно получать доступ напрямую с помощью сигналов через запись `categoryId = input.required<string>()` (взял решение из официальной документации). Здесь происходит какая-то магия: встроенный механизм смотрит на нейминг и при совпадении его с динамическим параметром автоматически подхватывает. Собственно, динамические айдишники нужны для моей реализации для сохранения местонахождения пользователя на сайте получения данных после перезагрузки страницы. Таким образом, реализован максимально удобный UX.

Однако остаются еще 2 проблемы, которые предстоит решить во второй половине недели:
Во-первых, отрисовать компонент хлебных крошек на маленьких экранах. Я посмотрел референсы и почитал стайлгайды на данную тему: рекомендуется уводить промежуточные крошки в многоточие, при клике на которых показывать поповером с выбором элементов. По времени может съесть пару часов, но такое решение показалось мне наиболее деликатным для пользователя. Рассматривал вариант просто переносить каждую хлебную крошку на следующую строку, но такой список в 4 уровня смотрится некрасиво и неудобно, занимает много места на экране телефона. Что крайне критично.
Во-вторых, нужно будет добавить динамическую отрисовку крошек на 2-м и 3-м уровнях вложенности (т.е. по названию категории и топика) - это тоже хороший UX. На моках оно работало, поскольку шло сопоставление по айдишнику с общим списком всех категорий и топиков. Но после перехода на api такое решение видится неоптимизированным: нет смысла запрашивать лишний раз данные с сервера, когда можно каким-то образом подтянуть данные в хлебную крошку из ответа, который приходит на текущем уровне (с этой целью мы доработали немного контракт с бэкендом).

2 и 3. Интеграция фронтенда квиза с бэкендом
В принципе, все прошло гладко. Предварительно составленный контракт в этом помог. Была архитектурная сложность. Изначально думал реализовать что-то в духе классического MVC, но в документации Angular предложено реактивное решение. По подходу и принципу работы оно похоже на Redux Toolkit: создается сервис, в котором хранится состояние (по сути, модель с данными) и вызовы ручек. Компоненты страниц, таким образом, остаются максимально "глупыми" и только получают/отправляют данные. Его и реализовал.
С точки зрения архитектуры всего проекта, это FSD на минималках: весь слой api хранится в моей фиче. Отсюда отпала необходимость в добавлении отдельного модуля Engine, который выступил бы, на мой взгляд, избыточной абстракцией над достаточно простой фичей. Предупредил об этом ребят, а главное - Никиту, который планировал его реализацию. Очень надеюсь, что сможем задействовать его ресурсы над реализацией других фич.
По интеграции в целом осталось исправить только 1 баг на бэкенде. Также следующими небольшими задачами я прорабатываю состояние загрузки страницы и кнопки, которые интегрирую в свой модуль.

Планировал интегрировать также библиотеку `tsparticles/angular` на финальный экран квиза с результатами, но решил пока поставить на паузу. Потребуется время для погружения в библиотеку, которое лучше задействовать для реализации более важных чем задач, чем декоративные эффекты.
